# Lessons Learned

## Issues Encountered and How They Were Resolved

### Issue 1 — Azure Monitor Agent Not Visible in Extensions Marketplace

**Problem:** When searching for AMA in the VM Extensions marketplace, it did not appear as an installable extension.

**Root cause:** Microsoft removed AMA from the VM Extensions marketplace. It is now installed automatically when a VM is associated with a Data Collection Rule.

**Resolution:** Went to Monitor → Data Collection Rules → DCR → Resources tab → added the VM. Azure automatically installed AzureMonitorWindowsAgent v1.43.0.0 within 5-10 minutes.

**Lesson:** Always check current Microsoft documentation for AMA installation methods as the process has changed significantly from older tutorials.

### Issue 2 — SecurityEvent Table Returning No Results

**Problem:** After setting up the full pipeline, SecurityEvent queries returned no results. Sysmon events were flowing correctly.

**Root cause:** The DCR was configured with a custom XPath rule that only collected Sysmon events. The Windows Security event log was never added as a data source.

**Resolution:** Added Security!* as a second XPath rule in the DCR data source. Security events began flowing within 10 minutes.

**Lesson:** When using custom XPath mode in a DCR, you must explicitly name every log channel you want to collect. Always verify data is flowing with a KQL query before building detections on top.

### Issue 3 — Incidents Not Appearing in the Defender Portal

**Problem:** Analytics rules were running and generating alerts but the Incidents page showed 0 incidents.

**Root cause:** The Defender portal applies a default Priority score: 15-100 filter to the Incidents view, which was hiding all incidents generated by custom Sentinel rules.

**Resolution:** Removed the Priority score filter from the Incidents page. 42 incidents immediately appeared.

**Lesson:** Always check active filters when a page appears empty in the Defender portal. Default filters can hide legitimate incidents from custom analytics rules.

### Issue 4 — Sentinel Moved to Defender Portal

**Problem:** Navigating to Microsoft Sentinel in portal.azure.com showed features had moved to the Defender portal. Menu items and navigation paths differed from documentation.

**Root cause:** Microsoft is migrating Sentinel into the unified Microsoft Defender portal. As of 2026, most Sentinel features are accessed via security.microsoft.com.

**Resolution:** Used the Defender portal for all Sentinel operations. The Logs query editor remains accessible via both portals.

**Lesson:** When following SC-200 study materials or older lab guides, expect portal navigation to differ. Always check current Microsoft documentation.

### Issue 5 — Playbook Authentication Failing Across Tenants

**Problem:** When opening the Logic App editor from the Defender portal, a 401 tenant mismatch error appeared. The playbook ran but failed due to connection authentication issues.

**Root cause:** The account existed in two different Azure tenants. The Defender portal was logged into a different tenant than the one containing the Azure subscription.

**Resolution:** Switched directory in the Azure portal to the correct tenant, then re-authenticated the Logic App connection by signing in again through the Change connection option in the designer.

**Lesson:** When working across the Defender portal and Azure portal, always verify you are in the correct directory. Tenant mismatches cause silent authentication failures that are difficult to diagnose without checking the error details carefully.

### Issue 6 — Playbook Not Appearing in Defender Portal Automation Rules

**Problem:** When creating an automation rule in the Defender portal, the playbook did not appear in the Run playbook dropdown.

**Root cause:** The Defender portal automation rules only support alert triggers, but the playbook was created with an incident trigger. Additionally, Sentinel needed explicit resource group permissions to run the playbook.

**Resolution:** Created the automation rule from the classic Sentinel portal (portal.azure.com) which supports incident triggers. Also configured playbook permissions via Sentinel Settings → Playbook permissions → granting access to rg-soc-lab.

**Lesson:** For incident-triggered playbooks, use the classic Sentinel portal to create automation rules. The Defender portal automation rules currently only support alert triggers.

## What I Would Do Differently

1. Plan DCR data sources before deployment — decide upfront which log channels to collect
2. Verify each pipeline stage before moving to the next — confirm AMA heartbeat before building DCR, confirm SecurityEvent data before building analytics rules
3. Use a smaller VM size from the start — Standard D2s v3 is more expensive than needed, B2s would suffice
4. Enable audit policies as part of VM setup rather than as a separate step
5. Read current Microsoft documentation rather than older tutorials for accurate portal navigation
